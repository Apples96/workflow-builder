WORKFLOW UGAP DC4 - CONTRÔLE AUTOMATISÉ V5 (Langage naturel pour non-développeurs avec validation SIRET via API externe)

=== CONTEXTE ===

L'utilisateur fournit 5 ou 6 documents en pièces jointes dans cet ordre précis :

1. **Document DC4** (fichier Word ou PDF)
2. **Avis d'appel public à la concurrence** (fichier PDF - peut être JOUE ou BOAMP)
3. **Acte d'engagement** (fichier PDF ou Word)
4. **Déclaration du candidat** (fichier PDF)
5. **Relevé d'identité bancaire (RIB)** (fichier PDF)
6. **[OPTIONNEL] DC4 initial** (fichier Word ou PDF - précédente déclaration de sous-traitance si le DC4 est un document rectificatif)

IMPORTANT : Les documents seront traités dans l'ordre où ils sont fournis.

=== PRINCIPES GÉNÉRAUX D'EXTRACTION ===

Pour tous les contrôles :
- Extraire d'abord les informations de chaque document
- Comparer ensuite les informations extraites
- Accepter les petites variations de format (majuscules/minuscules, espaces, ponctuation)
- Considérer deux textes comme identiques s'ils ont le même sens, même si la formulation varie légèrement

=== CLARIFICATIONS IMPORTANTES ===

**NUMÉRO DE RÉFÉRENCE DU MARCHÉ** :
Il existe plusieurs types de numéros dans les documents d'appel d'offres UGAP :
1. **Numéro de procédure** : Référence administrative UGAP (ex: 22U012) - apparaît généralement en haut des documents
2. **Numéro de marché** : Numéro attribué après signature du contrat (ex: 617529) - peut apparaître après la signature
3. **Code CPV** : Code de classification européen (8 chiffres commençant par 7 ou 4, ex: 72000000) - NE PAS confondre avec le numéro de marché
4. **SIRET** : Numéro d'identification d'entreprise (14 chiffres) - NE PAS confondre avec le numéro de marché

Lors de l'extraction du "numéro de référence du marché", chercher :
- Un code alphanumérique court (4-8 caractères comme 22U012 ou 617529)
- Généralement situé dans les sections d'identification du marché
- EXCLURE les codes CPV (8 chiffres commençant par 7 ou 4)
- EXCLURE les SIRET (14 chiffres)

**VARIATIONS DES SECTIONS DANS LES DOCUMENTS** :
Les documents UGAP peuvent utiliser différents formats pour nommer les sections :
- Document DC4 : utilise "Section A", "Section B", etc. avec des titres descriptifs (ex: "Section A - Identification de l'acheteur")
- Avis d'appel public : utilise "Section I.1", "Section II.1.1", etc. (ex: "Section I.1 Nom et adresses")
- Acte d'engagement : peut utiliser "Section C" ou simplement des titres sans numérotation

Le système doit être flexible et accepter ces variations :
- "Section A" = "A -" = "A." = "SECTION A" = "section a" (toutes sont équivalentes)
- La présence ou l'absence de tirets, points, majuscules ne change pas le sens

**FORMAT DES DATES** :
- Format préféré : JJ/MM/AAAA (format français standard, ex: 15/03/2024)
- Formats alternatifs acceptables : "le XX mois AAAA" (ex: le 15 mars 2024), "XX/MM/AAAA", "JJ-MM-AAAA"
- Le système doit tenter de convertir les dates dans différents formats vers le format standard JJ/MM/AAAA
- Toujours vérifier la cohérence des dates (pas dans le futur, pas avant l'an 2000)

## ZONE A : Identification de l'acheteur (Contrôles n°1-2)

**Objectif** : Vérifier que l'acheteur public est identifié de la même manière dans le DC4 et dans l'avis public.

**Documents à utiliser** : DC4 (document 1) + Avis (document 2)

**Étape 1 - Extraction des informations du DC4** :
Dans la section A - Identification de l'acheteur, extraire le nom de l'acheteur public, son adresse complète, son téléphone et son email

**Étape 2 - Extraction des informations de l'Avis** :
Dans la section I.1 Nom et adresses, extraire le nom du pouvoir adjudicateur, son adresse, le téléphone et l'email du point de contact

**Étape 3 - Comparaison** :
Comparer les informations extraites en acceptant les petites différences :
- **Nom** : Les deux noms doivent désigner la même entité (accepter les abréviations comme Cie pour Compagnie, ignorer majuscules/minuscules, accepter présence/absence d'acronymes entre parenthèses)
- **Adresse** : Les adresses doivent correspondre (accepter rue vs r., avenue vs av., ignorer majuscules/minuscules, accepter variations de ponctuation comme tirets/espaces, accepter Cedex comme suffixe optionnel, accepter codes postaux proches si même ville)
- **Téléphone** : Les numéros doivent être identiques (accepter différents formats avec espaces, points ou préfixe international, ignorer les chiffres incomplets s'ils correspondent sur les premiers chiffres significatifs)
- **Email** : Les emails doivent être identiques en ignorant strictement les majuscules/minuscules (normaliser en minuscules avant comparaison)

**Résultat** :
- Si toutes les informations correspondent => **CONFORME**
- Si au moins une information diffère significativement => **NON CONFORME**

Dans le rapport, expliquer précisément quelle information pose problème si non conforme.

## ZONE B : Objet du marché public (Contrôles n°3-4)

**Objectif** : Vérifier que le marché public est décrit de la même manière dans le DC4 et dans l'avis.

**Documents à utiliser** : DC4 (document 1) + Avis (document 2)

**Étape 1 - Extraction des informations du DC4** :
Dans la section B - Objet du marché public, extraire le numéro de référence du marché (format alphanumérique comme 22U012 ou 617529, exclure les codes CPV commençant par 7 ou 4 chiffres), l'intitulé du marché et toute mention de lot avec numéro

**Étape 2 - Extraction des informations de l'Avis** :
Dans la section II.1.1, extraire le numéro de référence, l'intitulé du marché et identifier le lot correspondant si plusieurs lots existent

**Étape 3 - Comparaison** :
Comparer :
- **Numéro de référence** : Normaliser les deux numéros en supprimant tous les espaces, caractères spéciaux et en convertissant en majuscules, puis comparer. Les deux numéros normalisés doivent être strictement identiques. Exemples de correspondances valides : 22U011 = 22u011 = 22 U 011 = N°22U011 = 22-U-011. Un mismatch comme 22U011 vs 21U120 est une vraie non-conformité.
- **Intitulé** : Les deux descriptions doivent avoir le même sens (accepter les reformulations légères, différences de niveau de détail, et variations de formulation tant que le domaine d'activité est cohérent)
- **Lot** : Si le DC4 mentionne un numéro de lot (exemples : Lot 6, Lot n°6, lot 06), extraire uniquement le chiffre du lot. Dans l'avis, chercher la mention du même lot (peut être dans des sections différentes comme II.2.1 pour chaque lot). Normaliser les numéros de lot en supprimant les zéros non significatifs et les formatages. Exemples de correspondances valides : Lot 6 = Lot n°6 = Lot 06 = lot 6. Si le DC4 mentionne un lot, l'Avis DOIT mentionner le même numéro de lot pour être conforme.

**Résultat** :
- Si tout correspond => **CONFORME**
- Si le numéro diffère ou l'intitulé ne correspond pas => **NON CONFORME**

## ZONE C : Objet de la déclaration du sous-traitant (Contrôles n°5-8)

**Objectif** : Vérifier qu'une seule option est sélectionnée et que les dates sont cohérentes.

**Documents à utiliser** : DC4 (document 1) + éventuellement document 6 si fourni

**Étape 1 - Identifier l'option cochée** :
Dans la section C - Objet de la déclaration du sous-traitant, identifier quelle option est cochée entre Un acte spécial (première déclaration) et Un acte spécial modificatif (modification). Analyser visuellement les cases cochées (☑, ☒, X, ×) et le texte explicite. Si option 2 cochée, extraire la date de la précédente déclaration

**Étape 2 - Vérifier qu'une seule option est cochée** :
- Si **aucune case** n'est cochée => **NON CONFORME** (Contrôle n°5 échoué) - BLOQUANT
- Si **deux cases** sont cochées => **NON CONFORME** (Contrôle n°5 échoué) - BLOQUANT (rejeter le DC4)
- Si **une seule case** est cochée => Passer à l'étape suivante

**Étape 3 - Si l'option 2 est cochée (acte modificatif)** :
Chercher dans la même section la **date de la précédente déclaration de sous-traitance**. Cette date est généralement indiquée juste après le texte de l'option 2, au format Un acte spécial modificatif du [DATE] où DATE est au format JJ/MM/AAAA (ex: 15/03/2024) ou le XX mois AAAA (ex: le 15 mars 2024).

- Si **aucune date** n'est trouvée => **NON CONFORME** (Contrôle n°6 échoué)
- Si une **date est présente** :
  * Vérifier que la date est cohérente (pas dans le futur par rapport à aujourd'hui 2025-12-04, pas avant 2000)
  * Si la date semble valide (entre 2000 et aujourd'hui) => **CONFORME** (Contrôle n°7 réussi)
  * Si la date est incohérente (ex: dans 10 ans, ou avant 2000) => **NON CONFORME** (Contrôle n°7 échoué)

**Étape 4 - Si l'option 1 est cochée (première déclaration)** :
=> **CONFORME** (Contrôle n°8 réussi)

**Résultat global** :
- Si une seule case cochée et validations passent => **CONFORME**
- Sinon => **NON CONFORME**

## ZONE D : Identification du candidat ou titulaire (Contrôles n°9-10-11)

**Objectif** : Vérifier que le candidat/titulaire est identifié de la même manière dans le DC4 et dans l'acte d'engagement, et valider les informations via l'API externe du gouvernement français.

**Documents à utiliser** : DC4 (document 1) + Acte d'engagement (document 3) + API externe https://recherche-entreprises.api.gouv.fr

**Étape 1 - Extraction des informations du DC4** :
Dans la section D - Identification du candidat ou du titulaire, extraire le nom commercial, la dénomination sociale, l'adresse du siège social, l'email, le téléphone, la forme juridique et le SIRET (14 chiffres)

**Étape 2 - Extraction des informations de l'Acte d'engagement** :
Dans la section C - Engagement du candidat ou mention Pour le compte de, extraire le nom commercial, la dénomination sociale, l'adresse, l'email, le téléphone, la forme juridique et le SIRET

**Étape 3 - Comparaison champ par champ** :
Comparer chaque information et noter si elle est conforme ou non :
1. **Nom commercial** : Doit être identique ou très similaire (accepter SAS ABC = ABC SAS, ignorer majuscules/minuscules)
2. **Dénomination sociale** : Doit être identique (c'est le nom officiel légal, mais accepter majuscules/minuscules)
3. **Adresse** : Doit correspondre (accepter variations mineures comme rue vs r., ignorer majuscules/minuscules)
4. **Email** : Doit être identique en normalisant en minuscules avant comparaison (les emails sont case-insensitive selon RFC 5321)
5. **Téléphone** : Normaliser les deux numéros avant comparaison en supprimant TOUS les espaces, points, tirets, parenthèses et le préfixe +33 (remplacer par 0). Comparer uniquement les 10 chiffres significatifs. Si les deux numéros normalisés sont identiques => CONFORME.
6. **SIRET** : Normaliser en supprimant TOUS les espaces, points, tirets. Un SIRET doit faire exactement 14 chiffres. Comparer les deux SIRET normalisés.

**Résultat de la comparaison DC4 vs Acte d'engagement** :
- Si **TOUS** les champs correspondent => **CONFORME** (Contrôle n°9)
- Si **AU MOINS UN** champ diffère => **NON CONFORME** (Contrôle n°10)

Dans le rapport, lister précisément les champs qui ne correspondent pas.

**Étape 4 - Validation externe via API gouvernementale (Contrôle n°11)** :

IMPORTANT : Utiliser la méthode `validate_siret_info()` du ParadigmClient pour interroger l'API externe du gouvernement français.

Appeler `validate_siret_info()` avec les paramètres extraits du DC4 :
- commercial_name : nom commercial extrait du DC4
- social_name : dénomination sociale extraite du DC4
- siret : numéro SIRET (14 chiffres) extrait du DC4
- address : adresse du siège social extraite du DC4
- legal_form : forme juridique extraite du DC4

La méthode retourne une query formatée comparant les informations du DC4 avec celles de l'API externe.

Ensuite, utiliser `chat_completion()` avec `guided_choice=["CONFORME", "NON_CONFORME"]` et la query retournée pour valider la cohérence.

**Critères de validation externe** :
- **Nom commercial** : Au moins une des deux valeurs (celle du DC4 ou celle de l'API externe) doit correspondre
- **Dénomination sociale** : Au moins une des deux valeurs (celle du DC4 ou celle de l'API externe) doit correspondre
- **SIRET** : Doit correspondre strictement (14 chiffres identiques)
- **Adresse** : Doit correspondre avec tolérance pour variations mineures
- **Forme juridique** : Doit correspondre (ex: SAS, SARL, SA, etc.)

**Résultat de la validation externe** :
- Si toutes les informations sont cohérentes avec l'API externe => **CONFORME** (Contrôle n°11)
- Si au moins une information diffère significativement => **NON CONFORME** (Contrôle n°11)

Dans le rapport, afficher les informations comparées (DC4 vs API externe) pour justifier la décision.

## ZONE F : Nature des prestations sous-traitées (Contrôles n°14-15)

**Objectif** : Vérifier que les prestations décrites dans le DC4 correspondent à l'objet du marché dans l'acte d'engagement.

**Documents à utiliser** : DC4 (document 1) + Acte d'engagement (document 3)

**Étape 1 - Extraction du DC4** :
Dans le document DC4, chercher la section F - Nature et prix des prestations sous-traitées, sous-section Nature des prestations.
Extraire la description complète des prestations (peut être un texte de plusieurs lignes décrivant les travaux/services).

**Étape 2 - Extraction de l'Acte d'engagement** :
Dans l'acte d'engagement, chercher la section Objet de l'accord-cadre ou Objet du marché.
Extraire la description de l'objet du marché.

**Étape 3 - Comparaison** :
Comparer les deux descriptions :
- Les deux textes doivent décrire **le même type de prestations**

- Le sens général et le domaine d'activité doivent être cohérents
- Considérer comme conformes si les prestations sous-traitées sont un sous-ensemble logique de l'objet global du marché

**Résultat** :

## ZONE G : Prix des prestations sous-traitées (Contrôle n°16)

**Objectif** : Vérifier que les montants HT, TVA et TTC sont cohérents entre eux.

**Documents à utiliser** : DC4 (document 1) uniquement

**Étape 1 - Extraction des montants** :
Dans le document DC4, section F - Nature et prix des prestations sous-traitées, sous-section Prix, extraire :
- Le montant **HT** (Hors Taxes) en euros
- Le taux de **TVA** en pourcentage (exemple : 20%)
- Le montant **TTC** (Toutes Taxes Comprises) en euros

**Étape 2 - Vérification de la TVA** :
Vérifier que le taux de TVA indiqué est **20%** (c'est le taux standard en France).
Si le taux est différent (5,5%, 10%, etc.), le signaler dans le rapport mais ne pas rejeter automatiquement (certains marchés ont des taux spéciaux).

**Étape 3 - Vérification de la cohérence HT/TTC** :
Calculer le montant TTC attendu à partir du HT et de la TVA :
- **Formule** : TTC calculé = HT × (1 + TVA/100)
- Exemple : Si HT = 10 000 € et TVA = 20%, alors TTC devrait être 10 000 × 1,20 = 12 000 €

Comparer le TTC calculé avec le TTC déclaré dans le document :
- Si la différence est **inférieure ou égale à 1 euro** (tolérance pour arrondis) => **CONFORME**
- Si la différence est **supérieure à 1 euro** => **NON CONFORME** (Contrôle n°16 échoué)

Dans le rapport, afficher :
- Montant HT déclaré
- Taux TVA déclaré
- Montant TTC déclaré
- Montant TTC calculé
- Écart entre les deux

**Résultat** :
- Si cohérence validée => **CONFORME**
- Si incohérence > 1 euro => **NON CONFORME**

## ZONE H : Conditions de paiement - IBAN (Contrôles n°17-20)

**Objectif** : Vérifier que l'IBAN est valide et identique dans le DC4 et le RIB.

**Documents à utiliser** : DC4 (document 1) + RIB (document 5)

**Étape 1 - Extraction de l'IBAN du DC4** :
Dans la section H - Conditions de paiement sous-section Compte à créditer, extraire l'IBAN complet (27 caractères français commençant par FR)

**Étape 1bis - Nettoyage robuste de l'IBAN extrait** :
Avant toute validation, nettoyer l'IBAN extrait en appliquant ces opérations dans l'ordre :
1. Supprimer **TOUS les types d'espaces** : espaces normaux, espaces insécables, tabulations, retours à la ligne, espaces fins, espaces cadratin, et tous autres caractères d'espacement Unicode
2. Convertir en majuscules
3. Ne garder que les lettres (A-Z) et les chiffres (0-9)
4. Vérifier que le résultat fait bien 27 caractères exactement

ATTENTION CRITIQUE : Il est absolument essentiel de supprimer TOUS les types d'espaces et caractères invisibles, pas seulement les espaces standards. Utiliser une méthode qui supprime tous les caractères d'espacement Unicode, car le document PDF peut contenir des espaces insécables ou d'autres caractères d'espacement invisibles qui feraient échouer la validation de longueur.

**Étape 2 - Validation de la structure de l'IBAN nettoyé** :
Vérifier que l'IBAN nettoyé respecte le format français :
1. **Longueur** : Doit contenir exactement 27 caractères
2. **Début** : Doit commencer par "FR" suivi de 2 chiffres (la clé de contrôle)
3. **Caractères** : Après "FRxx", il doit y avoir exactement 23 caractères (chiffres et lettres uniquement)

**Étape 3 - Validation mathématique de l'IBAN (algorithme modulo 97)** :
Pour vérifier que l'IBAN est vraiment valide (pas une erreur de frappe), appliquer la validation standard internationale :
1. Prendre l'IBAN sans espaces (exemple : FR7630006000011234567890189)
2. Déplacer les 4 premiers caractères (FR76) à la fin : 30006000011234567890189FR76
3. Remplacer chaque lettre par un nombre (F=15, R=27 car A=10, B=11, ..., Z=35)
4. Calculer si le nombre obtenu, divisé par 97, donne un reste de 1

Si le reste = 1, l'IBAN est **valide**. Sinon, il est **invalide**.

**Résultat de la validation** :
- Si l'IBAN est valide => **CONFORME** (Contrôle n°18)
- Si l'IBAN est invalide => **NON CONFORME** (Contrôle n°17) - **BLOQUANT** (rejeter le DC4 immédiatement)

**Étape 4 - Extraction de l'IBAN du RIB** :
Identifier l'IBAN dans le RIB (généralement dans un encadré ou ligne dédiée) ou reconstruire l'IBAN à partir des composants (code banque 5 chiffres, code guichet 5 chiffres, numéro compte 11 chiffres, clé RIB 2 chiffres)

**Étape 4bis - Nettoyage robuste de l'IBAN du RIB** :
Appliquer exactement le même nettoyage robuste que pour l'IBAN du DC4 (voir Étape 1bis) :
1. Supprimer **TOUS les types d'espaces** (espaces normaux, insécables, tabulations, retours à la ligne, etc.)
2. Convertir en majuscules
3. Ne garder que les lettres (A-Z) et les chiffres (0-9)
4. Vérifier que le résultat fait bien 27 caractères exactement

**Étape 5 - Comparaison des deux IBAN nettoyés** :
Comparer les deux IBAN **après avoir appliqué le nettoyage robuste à chacun** :
- Si les deux IBAN nettoyés sont **strictement identiques** (même séquence de 27 caractères) => **CONFORME** (Contrôle n°20)
- Si les deux IBAN nettoyés sont **différents** => **NON CONFORME** (Contrôle n°19) - **BLOQUANT** (rejeter le DC4)

Dans le rapport, afficher les deux IBAN côte à côte pour visualiser la différence si non conforme.

**Résultat global** :
- IBAN valide ET identique au RIB => **CONFORME**
- Sinon => **NON CONFORME** et DC4 rejeté

## ZONE L : Acceptation et agrément - Signatures (Contrôles n°21-22)

**Objectif** : Vérifier que les personnes qui signent le DC4 sont bien habilitées à le faire.

**Documents à utiliser** : DC4 (document 1) + Déclaration du candidat (document 4)

**Étape 1 - Identifier les signataires dans le DC4** :
Dans le document DC4, chercher la section L - Acceptation et agrément des conditions de paiement.
Cette section contient normalement 3 zones de signature :
1. Signature du **sous-traitant**
2. Signature du **titulaire** (ou soumissionnaire)
3. Signature de l'**acheteur** (à ignorer pour nos contrôles)

Pour chaque signature (sous-traitant et titulaire), extraire le nom et prénom de la personne qui signe.
Formats possibles : Dupont Jean, Jean DUPONT, M. Jean Dupont, Mme Dupont Jean.

**Étape 2 - Trouver la personne habilitée pour le sous-traitant** :
Dans le document DC4, section E - Identification du sous-traitant, chercher la mention Personne(s) physique(s) ayant le pouvoir d'engager le sous-traitant ou Représentant légal.
Extraire le nom et prénom de cette personne.

**Étape 3 - Vérifier la signature du sous-traitant (Contrôle n°21)** :
Comparer :
- Le nom du signataire dans la section L (signature du sous-traitant)
- Le nom de la personne habilitée dans la section E

Critères de correspondance :
- Le **nom de famille** doit être identique (ignorer majuscules/minuscules)
- Le **prénom** doit être identique OU au moins l'initiale doit correspondre (accepter Jean = J.)

**Résultat** :
- Si les noms correspondent => **CONFORME** (Contrôle n°21)
- Si les noms sont différents => **NON CONFORME** (Contrôle n°21)

**Étape 4 - Trouver la personne habilitée pour le titulaire** :
Dans la Déclaration du candidat (document 4), chercher la section C1 ou Personne(s) ayant le pouvoir d'engager la société ou Représentant légal.
Extraire le nom et prénom de cette personne.

**Étape 5 - Vérifier la signature du titulaire (Contrôle n°22)** :

## DÉCISION GLOBALE

Après avoir effectué tous les contrôles, calculer le statut global du DC4 :

### Contrôles BLOQUANTS (rejettent immédiatement le DC4) :
- **Contrôle n°5** : Deux cases cochées en Zone C => DC4 REJETÉ
- **Contrôle n°17** : IBAN invalide (mauvaise structure) => DC4 REJETÉ
- **Contrôle n°19** : IBAN DC4 différent de l'IBAN RIB => DC4 REJETÉ

Si au moins un contrôle bloquant échoue => **DC4 REJETÉ** immédiatement

### Contrôles CRITIQUES (maximum 3 échecs tolérés) :
- Contrôles Zone A (n°1-2) : Identification acheteur
- Contrôles Zone D (n°9-10-11) : Identification candidat/titulaire et validation API externe
- Contrôles Zone E (n°12-13) : Identification sous-traitant et validation API externe
- Contrôles Zone L (n°21-22) : Signatures

Si 4 contrôles critiques ou plus échouent => **DC4 REJETÉ**

### Contrôles STANDARD (signalent des anomalies mais ne bloquent pas) :
- Contrôles Zone B (n°3-4) : Objet du marché
- Contrôles Zone F (n°14-15) : Nature des prestations
- Contrôle Zone G (n°16) : Cohérence des prix

Ces contrôles sont signalés dans le rapport mais n'entraînent pas le rejet automatique.

### Décision finale :
1. S'il y a 1 ou plusieurs contrôles BLOQUANTS non conformes => **DC4 REJETÉ**
2. S'il y a 4 contrôles CRITIQUES ou plus non conformes => **DC4 REJETÉ**
3. Sinon => **DC4 ACCEPTÉ** (éventuellement avec réserves si des contrôles standard ont échoué)
