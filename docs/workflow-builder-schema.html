1. Architecture Globale du Système

🏗️ Vue d'Ensemble

Application web de création et d'exécution de workflows automatisés utilisant l'IA pour générer du code Python et l'API Paradigm pour le traitement de documents.

👤 Frontend

Interface utilisateur web

  • React/Next.js
  • Appels REST API
  • Upload de fichiers

⚙️ Backend API

FastAPI (Python)

  • Gestion workflows
  • Exécution code
  • Génération packages

💾 Stockage

Persistance des données

  • Redis
  • Stockage persistant
  • TTL: 24 heures

🤖 Services Externes

APIs IA et Documents

  • Anthropic Claude
  • LightOn Paradigm
🔑 Caractéristiques clés :
  • Conteneurisé Docker : Déployé avec Docker (mono-container)
  • Architecture monolithique : Application tout-en-un (Backend + Frontend)
  • Pas de base de données traditionnelle : Stockage en mémoire (Redis optionnel pour serverless)
  • Code généré par IA : Claude génère du code Python autonome

  • id : UUID généré automatiquement

Claude enrichit la description avec :

  • Détails techniques des étapes
  • Clarifications sur les APIs à utiliser
  • Précisions sur les formats de données
  • Questions de clarification
  • Avertissements sur les limitations
4 Génération du Code Python
Appel à Claude API avec :
  • Description enhanced du workflow
  • Documentation complète de l'API Paradigm
  • Exemples de code
  • Templates de fonctions
  • Gestion des erreurs et timeouts
# Exemple de code généré par Claude async def workflow_function( user_input: str, attached_file_ids: List[int] ): """ Analyse de factures et extraction de montants """ # Initialisation client Paradigm paradigm_client = ParadigmClient( api_key=LIGHTON_API_KEY, base_url="https://paradigm.lighton.ai" ) # Step 1: Recherche dans les documents search_response = await paradigm_client.document_search( query=user_input, file_ids=attached_file_ids, tool="DocumentSearch" ) # Step 2: Extraction des données structurées results = [] for doc in search_response["documents"]: extracted = extract_amounts(doc["chunks"]) results.append(extracted) # Step 3: Formatage du résultat return format_html_table(results)

4. Upload et Traitement de Fichiers

📁 Upload vers Paradigm API

Les fichiers sont uploadés vers l'API Paradigm qui se charge de les parser, chunker et indexer.

Flux d'Upload et Indexation

1 Upload Fichier
POST /api/files/upload
multipart/form-data: - file: facture.pdf - collection_type: "private"
2 Envoi à Paradigm
POST https://paradigm.lighton.ai/api/v2/files
3 Traitement Asynchrone par Paradigm

a) Parsing

  • Extraction du texte
  • OCR si nécessaire
  • Extraction métadonnées

b) Chunking

  • Découpage sémantique
  • ~500 tokens/chunk
  • Préservation du contexte

c) Embeddings

  • Vectorisation chunks
  • Modèle LightOn
  • Dimension: 1024

d) Indexation

  • Stockage SQL
  • Index vectoriel
  • Métadonnées
4 Statut du Fichier
États possibles :
  • processing : En cours de traitement
  • embedded : Prêt, indexé et recherchable ✅
  • failed : Échec du traitement ❌
5 Réponse
{ "id": 123, "filename": "facture.pdf", "status": "embedded", "bytes": 245678, "created_at": 1701518400 }

🗄️ Stockage dans Paradigm (Base de données SQL)

Table Champs Principaux Description
files id, filename, status, collection_type, owner_id Métadonnées des fichiers uploadés
chunks id, file_id, text, embedding, position Morceaux de texte avec vecteurs d'embedding

5. Workflow Runner - Exécution

🚀 Fichier : api/workflow/executor.py

Exécute le code Python généré dans un environnement sandbox sécurisé.

Flux d'Exécution

1 Requête d'Exécution
POST /api/workflows/{workflow_id}/execute { "user_input": "Extraire les montants totaux", "attached_file_ids": [123, 124, 125] }
2 Récupération du Workflow depuis Redis
# Connexion au service Redis via réseau Docker redis_client = redis.Redis( host="redis", # Nom du service Docker port=6379, decode_responses=True ) # Récupération du workflow workflow = redis_client.get(f"workflow:{workflow_id}")
Si le workflow n'existe pas ou a expiré (TTL dépassé) → 404 Not Found
3 Vérification des Fichiers Attachés
Polling des fichiers (max 60 secondes) :
for file_id in attached_file_ids: for attempt in range(30): # 30 × 2s = 60s status = paradigm_client.get_file_status(file_id) if status == "embedded": break # Fichier prêt ✅ elif status == "failed": raise Error("File processing failed") ❌ await asyncio.sleep(2) # Attendre 2 secondes
4 Création WorkflowExecution
execution = WorkflowExecution( id="xyz-789-uvw", workflow_id="abc-123-def", user_input="Extraire les montants totaux", status=ExecutionStatus.PENDING, created_at=now() ) # Stockage en mémoire uniquement (éphémère) WorkflowExecutor.executions[execution_id] = execution
5 Préparation Environnement Sandbox

🔐 Injection Clés API

# Remplacement dans le code LIGHTON_API_KEY = "sk-live-..." ANTHROPIC_API_KEY = "sk-ant-..."

🌐 Contexte Global

global_context = { "user_input": "...", "attached_file_ids": [...] }

🛡️ Sécurité

  • Built-ins restreints
  • Pas d'accès filesystem
  • Timeout: 20 minutes
6 Exécution du Code Généré
# Mise à jour du statut execution.status = ExecutionStatus.RUNNING start_time = time.time() # Exécution dans le sandbox try: result = await execute_in_sandbox( code=workflow.generated_code, context=global_context, timeout=1200 # 20 minutes ) execution.status = ExecutionStatus.COMPLETED execution.result = result except Exception as e: execution.status = ExecutionStatus.FAILED execution.error = str(e) finally: execution.execution_time = time.time() - start_time execution.completed_at = now()
7 Réponse à l'Utilisateur
{ "workflow_id": "abc-123-def", "execution_id": "xyz-789-uvw", "status": "COMPLETED", "result": "<table>...</table>", "execution_time": 8.234, "created_at": "2025-12-02T10:15:00Z", "completed_at": "2025-12-02T10:15:08Z" }

Les résultats d'exécution peuvent être téléchargés sous forme de rapport PDF professionnel :

GET /api/workflows/{workflow_id}/executions/{execution_id}/pdf
  • 📊 Rapport formaté avec résultats

6. Recherche Vectorielle (Paradigm API)

🔍 Document Search - Recherche Sémantique

Pendant l'exécution du workflow, le code généré peut interroger les documents via la recherche vectorielle.

Processus de Recherche Sémantique

1 Requête du Code Workflow
POST /api/v2/chat/document-search
{ "query": "Quel est le montant total?", "file_ids": [123, 124, 125], "tool": "DocumentSearch" }
2 Vectorisation de la Query

La question est transformée en vecteur d'embedding (1024 dimensions) :

"Quel est le montant total?" ↓ embedding_query = [0.31, -0.08, 0.81, 0.22, ...]
3 Recherche de Similarité

Calcul de similarité cosinus pour chaque chunk

Chunk Fichier Texte Score Sélectionné
460 facture_jan.pdf "Total TTC: 1,200.00 EUR" 0.95 ✅
458 facture_jan.pdf "Total HT: 1,000.00 EUR" 0.92 ✅
312 facture_feb.pdf "TVA 20%: 200.00 EUR" 0.88 ✅
101 facture_jan.pdf "Facture n° 12345..." 0.45 ❌
4 Génération de la Réponse par LLM

Le LLM Paradigm synthétise une réponse basée sur les chunks les plus pertinents :

Prompt au LLM: ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ Question: Quel est le montant total? Contexte (documents pertinents): - Total TTC: 1,200.00 EUR - Total HT: 1,000.00 EUR - TVA 20%: 200.00 EUR Réponds en français en citant les sources. ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ Réponse générée: "Le montant total TTC est de 1,200.00 EUR, composé d'un montant HT de 1,000.00 EUR et d'une TVA de 200.00 EUR (20%)."
5 Réponse Complète
{ "answer": "Le montant total TTC est de 1,200.00 EUR...", "documents": [ { "id": 123, "title": "facture_jan.pdf", "chunks": [ { "id": 460, "uuid": "3fa85f64-5717-4562-b3fc-2c963f66afa6", "text": "Total TTC: 1,200.00 EUR", "score": 0.95 }, { "id": 458, "text": "Total HT: 1,000.00 EUR", "score": 0.92 } ] }, { "id": 124, "title": "facture_feb.pdf", "chunks": [...] } ] }

7. Génération de Package Standalone

📦 Fichier : api/workflow/package_generator.py

Génère deux types de packages : Workflow Runner (application web complète) et MCP Server (serveur Model Context Protocol pour intégration LightOn Paradigm en priorité, et Claude Desktop comme solution temporaire).

🎯 Deux Types de Packages Disponibles :

📊 Workflow Runner Package

Application web standalone avec frontend personnalisé et backend intégré

  • Interface utilisateur HTML/CSS/JS
  • API FastAPI backend
  • Déployable indépendamment

🔌 MCP Server Package

Serveur Model Context Protocol pour intégration Paradigm (cible) et Claude Desktop (temporaire)

  • Mode HTTP pour Paradigm (Render)
  • Mode stdio pour Claude Desktop (local)
  • Tools exposés via MCP

Flux de Génération de Package

1 Analyse du Code Workflow

Claude analyse le code généré pour identifier :

  • Types de paramètres d'entrée (texte, fichiers, etc.)
  • Structure des résultats
  • APIs Paradigm utilisées
  • Configuration UI nécessaire
  • Paramètres pour tools MCP
2 Choix du Type de Package
L'utilisateur choisit le type de package à générer via l'API :
GET /api/workflows/{workflow_id}/download?package_type=workflow_runner # ou GET /api/workflows/{workflow_id}/download?package_type=mcp_server

  • Serveur FastAPI

4 Génération MCP Server Package
🔌 Deux modes de déploiement MCP Server :

Le MCP Server peut être déployé de deux manières :

  • Mode 1 - HTTP (LightOn Paradigm) - 🎯 Architecture cible, déployé sur URL publique (Render) | ⚠️ Bug temporaire sur transmission des file_ids
  • Mode 2 - stdio (Claude Desktop) - ✅ Fonctionne parfaitement en local, solution temporaire pendant correction du bug

Les deux modes utilisent uniquement Paradigm API - pas de Claude API externe.

Une fois le bug résolu, le Mode 1 (Paradigm HTTP) deviendra la solution principale recommandée.

🔌 Composants du MCP Server

Le MCP Server expose le workflow comme un "tool" utilisable par Claude Desktop via le Model Context Protocol.

server.py

  • Serveur MCP stdio
  • Pour Claude Desktop local
  • Communication bidirectionnelle

http_server.py

  • Serveur MCP HTTP
  • Pour Paradigm (déployé sur Render)
  • URL publique avec endpoint /mcp

workflow.py

  • Code workflow généré
  • Utilise Paradigm API uniquement
  • Exécution autonome

paradigm_client.py

  • Client API Paradigm standalone
  • Gestion retry
  • Documentation complète

🔄 Deux Modes de Déploiement

🎯 Mode 1 : MCP Server HTTP (LightOn Paradigm) - Architecture Cible

Fonctionnement :

  1. Serveur MCP HTTP (http_server.py) déployé sur Render.com
  2. URL publique accessible : https://app-name.onrender.com/mcp
  3. Paradigm communique avec le serveur via HTTP
  4. Le workflow utilise Paradigm API directement
  5. Les résultats sont retournés à Paradigm

Déploiement sur Render :

# Build Command : pip install mcp aiohttp python-dotenv uvicorn # Start Command : python -m http_server --host 0.0.0.0 --port 10000 # Variables d'environnement : PARADIGM_API_KEY=sk-... PARADIGM_BASE_URL=https://paradigm.lighton.ai MCP_BEARER_TOKEN=votre-token-securise

Configuration dans Paradigm (Admin) :

  • Admin > MCP Servers > Add MCP Server
  • Name : workflow-name
  • URL : https://app-name.onrender.com/mcp
  • Bearer Token : (le même que MCP_BEARER_TOKEN)

Avantages (une fois le bug résolu) :

  • ✅ Pas de timeout de 4 minutes
  • ✅ Peut traiter beaucoup de documents
  • ✅ Accessible depuis n'importe où
  • ✅ Intégré dans l'interface Paradigm
  • ✅ Mode conversationnel natif avec Agent Paradigm
⚠️ Bug Connu (Temporaire) :
  • ❌ Les file_ids uploadés via Paradigm ne sont pas transmis au workflow
  • Le workflow reçoit des file_ids vides ou incorrects
  • Bug en cours d'investigation avec l'équipe Paradigm
  • Workaround temporaire : Utilisez le Mode 2 (Claude Desktop)
  • Une fois résolu : Ce mode deviendra la solution principale recommandée
✅ Mode 2 : MCP Server Local (Claude Desktop) - Solution Temporaire

Fonctionnement :

  1. Serveur MCP stdio (server.py) tourne en local
  2. Claude Desktop communique avec le serveur via stdin/stdout
  3. Le workflow utilise Paradigm API directement
  4. Les résultats sont retournés à Claude Desktop via MCP protocol

Configuration :

{ "mcpServers": { "workflow-name": { "command": "py", "args": ["-3.10", "-m", "server"], "cwd": "C:\\path\\to\\mcp-server" } } } # Fichier .env requis : PARADIGM_API_KEY=sk-... PARADIGM_BASE_URL=https://paradigm.lighton.ai

Avantages :

  • ✅ Fonctionne parfaitement actuellement
  • Latence minimale (local)
  • Pas de serveur à déployer
  • Utilise chemins absolus des fichiers locaux
  • Contournement efficace du bug Paradigm

Limitations :

  • ⏱️ Timeout de 4 minutes (limite Claude Desktop)
  • Recommandé pour 3-5 documents maximum
  • Exécution locale uniquement (pas d'accès distant)
  • Solution temporaire : À utiliser en attendant la résolution du bug Paradigm
📋 Architecture Commune :

Les deux modes partagent la même architecture backend :

  • ✅ Utilisent uniquement Paradigm API (pas de Claude API)
  • ✅ Même code workflow généré (workflow.py)
  • ✅ Même client Paradigm (paradigm_client.py)
  • ⚙️ Seul le transport change : stdio vs HTTP

Endpoints Principaux

POST /api/v2/chat/document-search

Recherche sémantique dans documents

  • query (requis)
